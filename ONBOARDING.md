# Onboarding: from a prompt to a finished game

This is the procedure for turning a one-paragraph concept into a complete, verified weft game
— by hand, or by handing it to a capable LLM (e.g. Opus). The engine is designed so that the
*only* required reading to author a game is `AUTHORING.md` (format) and `WRITING.md` (prose),
and so that correctness is **machine-checkable**: a generated game is not "done" until
`weft all` is green.

## The interview (decisions to pin down first)

Before writing scenes, settle these. If a human is onboarding, ask them; if an LLM is
generating from a bare prompt, choose sensible defaults and state them.

1. **Premise** — one paragraph: world, protagonist, the central tension, the kind of ending
   space (redemption? escape? mastery? tragedy?).
2. **Length** — scene count target (a tight game is 12–20 scenes; a book is 100+). Endings
   count (3 is a good minimum).
3. **Systems** — which to enable:
   - *Pure narrative* — flags + skill checks only. (No `systems.combat`, no `enemies`.)
   - *+ Inventory / clues* — items, gated/hidden choices.
   - *+ Stats & checks* — `check(stat, dc)` branching.
   - *+ Combat* — `systems.combat`, techniques, enemies, stances, bosses.
4. **Voice** — POV/tense and any signature phrases or motif words the narrator owns. Encode in
   `games/<id>/writing.js`. (Read `WRITING.md`.)
5. **Art style** — pick a preset for `meta.art.style` (`ink-wash`, `noir`, `storybook`,
   `pixel`, `oil`) or supply a custom `descriptor` + `palette`. This sets BOTH the
   illustration prompts and the UI theme.
6. **Surfaces** — which control panels the game needs (`meta`/`surfaces`): a journal ("the
   story so far"), a character sheet, an inventory/"satchel". Decide labels.

## The procedure

```
# 1. Scaffold a passing skeleton.
node tools/cli.mjs new games/<id> <id> "Title"

# 2. Edit games/<id>/game.js — set meta, stats, pools, items, enemies, systems, surfaces,
#    meta.art.style. Declare EVERY flag you will read in state.vars.

# 3. Write games/<id>/scenes/*.dsl following AUTHORING.md and WRITING.md.
#    Use `art: <slot>` and `brief: <subject>` on scenes you want illustrated.
#    Record milestones with chronicle('...') in choice `do:` blocks.

# 4. Add games/<id>/replays.json — one golden-path spec per ending.

# 5. (optional) games/<id>/writing.js — voice config / signatures / motifs.

# 6. Verify. Iterate until green.
node tools/cli.mjs all games/<id>      # compile + audit + lint(gate) + test
node tools/cli.mjs lint games/<id> --warn   # strict prose pass before locking

# 7. Art. Placeholders are generated by `build`; for real images:
node tools/cli.mjs art games/<id> --generate    # uses OPENROUTER_API_KEY from .env
```

## The acceptance gate (definition of done)

A game is finished only when **all** of the following hold — verify, do not assume:

- `weft all` prints `audit OK` and `tests OK`.
- **Audit:** 0 render errors, 0 unknown targets, 0 unreachable scenes.
- **Tests:** every replay PASSES; fuzz shows **100% coverage**, **0 softlocks**, **0 errors**,
  and **every ending appears** in the fuzz histogram.
- **Lint:** 0 errors; warnings reviewed and justified (`lint --warn` for the strict pass).
- Each ending has a replay spec. Each gated path is reachable in the fuzz histogram.

## Why this is LLM-friendly

- **One declarative format.** A game is data (`game.js`) + a line-oriented DSL — both easy to
  generate and diff.
- **Loud, specific failure.** The compiler rejects dead ends, bad jumps, undeclared variables
  (the silent-typo bug), and undeclared techniques. The audit proves reachability. The fuzzer
  finds softlocks and unreachable endings. The linter catches AI prose tells. Every failure is
  a precise, fixable message — the exact feedback loop an LLM needs to converge.
- **Determinism.** Seeded RNG means a generated replay spec is a permanent regression test.

> The two example games in `games/mystery` (pure narrative) and `games/crawl` (stats + combat
> + inventory) were authored from `AUTHORING.md` alone and pass the full gate. Use them as
> worked references.

## A ready-to-use generation prompt

> You are authoring a game for the **weft** interactive-fiction engine. Read `AUTHORING.md`
> and `WRITING.md` in full, and study `games/mystery` and `games/crawl` as references.
> Build the game described below under `games/<id>/`.
>
> CONCEPT: «one paragraph».
> SYSTEMS: «pure-narrative | +inventory | +stats/checks | +combat».
> LENGTH: «N scenes, M endings». VOICE: «POV/tense, any signatures». ART STYLE: «preset».
> SURFACES: «journal / inventory / etc.».
>
> Constraints: declare every flag you read in `state.vars`; mark true endings `ending:true`;
> all randomness via `check()/rand()`; obey `WRITING.md` (the linter gates it). Provide a
> `replays.json` reaching every ending. Run `node tools/cli.mjs all games/<id>` and iterate
> until it prints `audit OK` and `tests OK` with 100% fuzz coverage, 0 softlocks, 0 errors,
> and every ending in the histogram. Report the final command output.
